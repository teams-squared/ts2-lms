# Teams Squared LMS — Design System

This document is the authoritative reference for the Teams Squared LMS visual revamp. When making any UI change, check this document first. When in doubt, follow the principle in [Section 1](#1-design-thesis) and the rules in [Section 8](#8-component-rules).

> **How to use this document:** Read Section 1 to understand intent. Use Sections 2–7 as token and style references. Use Section 8 as prescriptive component rules. Use Section 10 as a migration checklist.

---

## 1. Design thesis

**Teams Squared LMS treats the brand purple as a signal color, not a wallpaper. Surfaces stay neutral so lessons can breathe. Purple marks what the learner should act on next.**

Three rules derive from this:

1. **Bold where it counts, quiet everywhere else.** Saturated brand purple (`#5006E3`) is used for: primary CTAs, active navigation states, progress fills, focus rings, key status badges. It is NOT used for: card backgrounds, hero washes, body text at scale, decorative elements.

2. **Solids over gradients in product UI.** Gradients are reserved for specific "moments" (login splash, completion celebration, certificate). Everywhere else uses solid color.

3. **One primary action per view.** If the page has a primary purple button, no other purple filled button appears in the same logical group. Secondary actions use outline or ghost styles.

When a visual decision conflicts with one of these rules, the decision is wrong. Re-approach it.

### 3.4 Color usage rules

| Use case | Token | Example class |
|---|---|---|
| Page background | `--background` | `bg-background` |
| Card surface | `--surface` | `bg-surface` |
| Selected nav item, subtle brand tint | `--surface-muted` / `--primary-subtle` | `bg-primary-subtle` |
| Primary button fill | `--primary` | `bg-primary text-primary-foreground` |
| Primary text on white (links, active labels) | `--primary` | `text-primary` |
| Body text | `--foreground` | `text-foreground` |
| Secondary text, meta | `--foreground-muted` | `text-foreground-muted` |
| Captions, timestamps | `--foreground-subtle` | `text-foreground-subtle` |
| Default border | `--border` | `border-border` |
| Input border, strong divider | `--border-strong` | `border-border-strong` |
| Focus ring | `--ring` | `ring-ring` |

**Forbidden:**
- `bg-[#5006E3]` or any hardcoded hex. Always use tokens.
- `bg-purple-600`, `text-indigo-500`, or any Tailwind default color. Always use semantic tokens.
- `bg-gradient-to-r from-purple-500 to-pink-500` or any gradient in product UI unless explicitly allowed in Section 8.10.
- Opacity-based dimming of brand color as a "hover" state. Use `--primary-hover` token.

**Dark-mode subtle parity:** Light-mode `-subtle` fills use low-chroma pastels with plenty of headroom. Mirror that headroom in dark mode by keeping chroma low (`≤ 0.08`) and lightness around `0.22`. An over-saturated dark-mode subtle (e.g., `oklch(0.25 0.10 25)`) reads as a colored wall rather than a quiet tint — the light/dark flavors must feel like the same family, only lit differently. When tuning a `*-subtle` token, check it against its partner by eye, not just by name.

## 4. Typography

### 4.1 Font families

```css
/* app/layout.tsx or globals.css */
--font-sans: 'Inter', ui-sans-serif, system-ui, -apple-system, sans-serif;
--font-display: 'Plus Jakarta Sans', 'Inter', ui-sans-serif, system-ui, sans-serif;
```

- **Body / UI:** Inter (variable font). Load via `next/font/google`.
- **Display / Headings:** Plus Jakarta Sans — closer to the Teams Squared wordmark's geometric feel. Use for `h1`, `h2`, marketing hero text, course card titles.
- **Wordmark:** Used only in the actual logo file. Do not attempt to match in UI text.

```tsx
// app/layout.tsx
import { Inter, Plus_Jakarta_Sans } from 'next/font/google';

const inter = Inter({ subsets: ['latin'], variable: '--font-sans' });
const jakarta = Plus_Jakarta_Sans({
  subsets: ['latin'],
  variable: '--font-display',
  weight: ['500', '600', '700'],
});
```

### 4.2 Type scale

Based on a 1.25 ratio, rounded for pixel-hinting:

| Token | Size | Line height | Use |
|---|---|---|---|
| `text-xs` | 12px | 1.5 | Captions, badges, timestamps |
| `text-sm` | 14px | 1.5 | Meta, secondary UI text |
| `text-base` | 16px | 1.6 | Default body |
| `text-lg` | 18px | 1.6 | Lesson reading body, emphasized body |
| `text-xl` | 20px | 1.5 | Card titles, small headings |
| `text-2xl` | 24px | 1.4 | h3, section headings |
| `text-3xl` | 30px | 1.3 | h2 |
| `text-4xl` | 36px | 1.2 | h1 |
| `text-5xl` | 48px | 1.1 | Display, hero |

### 4.3 Typography rules

- **Lesson reading content:** 17–18px, line-height 1.7. Learners spend long sessions reading; density hurts retention.
- **Headings use `font-display` (Jakarta), body uses `font-sans` (Inter).**
- **Weights used:** 400 (body), 500 (UI labels, emphasis), 600 (headings, button text), 700 (hero display only). No 300 or 900.
- **Never use `italic` for emphasis in UI text.** Italic is reserved for quotations and book/course titles in prose.
- **Line length:** Reading content caps at ~70 characters (`max-w-[65ch]`). UI text caps at whatever fits the container.
- **No all-caps except in badges, small labels (≤12px), and the 'squared' wordmark style.**

## 6. Radii

| Token | Value | Use |
|---|---|---|
| `rounded-sm` | 6px | Inputs, chips, small controls |
| `rounded-md` | 10px | Buttons, small cards |
| `rounded-lg` | 14px | Cards, modals, dropdowns |
| `rounded-xl` | 20px | Feature cards, hero panels |
| `rounded-full` | 9999px | Avatars, pill badges, progress bars |

**Rule:** Never mix three or more radii on the same visual tier. A card at `rounded-lg` contains buttons at `rounded-md` and inputs at `rounded-sm`. That's consistent. A card at `rounded-xl` with buttons at `rounded-sm` is not.

## 7. Elevation (shadows)

### 7.1 Light mode

```css
--shadow-sm: 0 1px 2px rgb(15 15 20 / 0.04);
--shadow-md: 0 4px 12px rgb(15 15 20 / 0.06), 0 2px 4px rgb(15 15 20 / 0.04);
--shadow-lg: 0 20px 40px rgb(15 15 20 / 0.10), 0 8px 16px rgb(15 15 20 / 0.06);
--shadow-xl: 0 32px 64px rgb(15 15 20 / 0.14), 0 16px 24px rgb(15 15 20 / 0.08);
```

### 7.2 Dark mode

Shadows are nearly invisible on dark backgrounds. Rely on borders and subtle surface elevation (lighter backgrounds for raised elements) instead of heavy shadows.

```css
.dark {
  --shadow-sm: 0 1px 2px rgb(0 0 0 / 0.4);
  --shadow-md: 0 4px 12px rgb(0 0 0 / 0.5), 0 2px 4px rgb(0 0 0 / 0.3);
  --shadow-lg: 0 20px 40px rgb(0 0 0 / 0.6), 0 8px 16px rgb(0 0 0 / 0.4);
}
```

### 7.3 Usage

| Level | Use |
|---|---|
| `shadow-sm` | Inputs at rest, subtle card lift |
| `shadow-md` | Course cards on hover, dropdowns, popovers |
| `shadow-lg` | Modals, dialogs |
| `shadow-xl` | Command palette, full-screen overlays |

## 8. Component rules

### 8.1 Buttons

**Three variants only.** If a fourth is needed, revisit the design instead of adding a variant.

| Variant | Appearance | Use |
|---|---|---|
| `primary` | `bg-primary text-primary-foreground` | Main CTA, one per logical group |
| `secondary` | `bg-background border border-border-strong text-foreground` | Secondary actions, alternatives to primary |
| `ghost` | `bg-transparent text-foreground hover:bg-surface-muted` | Tertiary actions, toolbar buttons |
| `destructive` | `bg-danger text-danger-foreground` | Delete, unenroll, irreversible actions |

**Sizes:**

| Size | Height | Padding | Use |
|---|---|---|---|
| `sm` | 32px | `px-3 py-1.5` | Table row actions, inline actions |
| `md` (default) | 40px | `px-4 py-2.5` | Standard |
| `lg` | 48px | `px-6 py-3` | Hero CTAs, landing, sign-up |

**States (required for every button):**

- `default` — as specified above
- `hover` — `bg-primary-hover` (primary), `bg-surface-muted` (ghost), border shifts to `--border-strong` (secondary)
- `active`/`pressed` — `bg-primary-active`, or translate-y-[1px] for tactile feel
- `focus-visible` — `ring-2 ring-ring ring-offset-2 ring-offset-background`
- `disabled` — `opacity-50 cursor-not-allowed`, no hover effects
- `loading` — spinner replaces text via `<Spinner />`, button maintains original width (use `min-w-*` or opacity trick on text)

**Minimum hit area:** 40×40px for all interactive buttons, per WCAG 2.5.5.

**Icon-only buttons** MUST have `aria-label`:
```tsx
<Button size="icon" aria-label="Open course menu">
  <Menu className="h-4 w-4" />
</Button>
```

### 8.2 Cards (course cards, lesson cards)

**Anatomy:**
```
┌──────────────────────────┐
│  [16:9 thumbnail]        │
├──────────────────────────┤
│  Category badge          │
│                          │
│  Course Title (2 lines)  │
│  Short description...    │
│                          │
│  6 modules · 2h 30m      │
│  ━━━━━━━━━━━░░░░  65%    │
└──────────────────────────┘
```

**Specs:**
- `bg-surface border border-border rounded-lg shadow-sm overflow-hidden`
- Thumbnail: `aspect-video` (16:9), object-cover
- Content padding: `p-4` (16px)
- Title: `text-lg font-semibold font-display line-clamp-2`
- Meta row: `text-sm text-foreground-muted flex items-center gap-3`
- Progress bar (if enrolled): Full width, 6px tall, `rounded-full`, track `bg-border`, fill `bg-primary`

**Hover:**
```css
transition: all 150ms ease-out;
&:hover {
  border-color: var(--border-strong);
  box-shadow: var(--shadow-md);
  transform: translateY(-2px);
}
```

**Accessibility:** The entire card must be a single clickable region — wrap in `<Link>` or `<button>`. Do NOT nest multiple tap targets (e.g., a "Bookmark" button inside a clickable card). If a secondary action is needed, place it OUTSIDE the card or use a separate detail page.

#### 8.2.1 Course thumbnails

All course thumbnails — catalog grid, dashboard rail, "you might also like" rows — go through the shared `<CourseThumbnail />` component. One component, one visual.

**With image:**
- 16:9 aspect ratio (`aspect-video`), `object-cover`
- Locked courses render the image at `grayscale` to cue state without extra chrome

**Without image (empty / fallback):**
- `bg-primary-subtle` fill
- Centered icon (`GraduationCap`, `h-9 w-9`) tinted `text-primary-subtle-foreground`
- Course title underneath, `text-sm font-medium text-primary`, `line-clamp-2`
- No gradients, no patterns — brand tint only. This is the one place the brand family shows up on cards.

**Do NOT** render a bespoke thumbnail inline. Always import `CourseThumbnail` from `@/components/courses/CourseThumbnail`. Changing the empty state in one place must change it everywhere.

The `bg-primary-subtle` container is applied in **both** cases — when a real image exists and when the fallback icon is shown. A course with a real thumbnail renders the image using `object-contain` with padding (`p-6`), centred within the tinted frame. This ensures visual consistency across cards regardless of whether individual courses have thumbnails assigned.

**Never** render a raw image on a white background within a card. The tinted frame is the constant. Using `object-cover` without padding can allow images with white or transparent backgrounds to fill the tile and obscure the tinted frame — defeating the consistency goal.

### 8.3 Modals / Dialogs

Use shadcn/ui `Dialog` (Radix primitive) as the base.

**Specs:**
- Backdrop: `bg-foreground/50 backdrop-blur-sm` (light), `bg-black/70 backdrop-blur-sm` (dark)
- Content: `max-w-lg` default, `bg-background border border-border rounded-lg shadow-lg p-6`
- Enter animation: fade + scale from `0.96` to `1`, 180ms ease-out
- Exit animation: 120ms ease-in

**Anatomy:**
```
┌──────────────────────────────┐
│  Title                    ✕  │  ← DialogHeader
│  Optional description        │
├──────────────────────────────┤
│                              │
│  [Body content]              │  ← DialogBody
│                              │
├──────────────────────────────┤
│         [Cancel] [Confirm]   │  ← DialogFooter, right-aligned
└──────────────────────────────┘
```

**Behavior:**
- Focus trap enabled (Radix handles this)
- Escape closes (except during multi-step flows or destructive actions mid-input)
- Click outside closes (same exception)
- Return focus to trigger element on close
- Do NOT nest modals. For multi-step flows, use a stepper pattern inside a single modal.

**Destructive actions (e.g., "Delete course", "Unenroll"):**
- Use `AlertDialog` (not `Dialog`) — confirm required before close
- Primary action button uses `variant="destructive"`
- Cancel is always the default focus target

### 8.4 Forms & inputs

**Input specs:**
- Height: 40px
- Padding: `px-3 py-2.5`
- Border: `border border-border-strong rounded-sm`
- Focus: `focus-visible:ring-2 focus-visible:ring-ring focus-visible:ring-offset-2 focus-visible:border-primary`
- Placeholder: `placeholder:text-foreground-subtle`
- Background: `bg-background`

**Label placement:**
```tsx
<div className="space-y-1.5">
  <Label htmlFor="email">Email address</Label>
  <Input id="email" type="email" />
  <p className="text-sm text-foreground-muted">We'll never share your email.</p>
</div>
```

**Rules:**
- Labels are ALWAYS visible above the input. Never use placeholder text as a label.
- Every input has an associated `<Label htmlFor>`.
- Required fields marked with a visible indicator (asterisk + `aria-required`), not color alone.

**Error state:**
- Border shifts to `border-danger`
- Helper text below shows error message in `text-danger`
- Include an error icon (e.g., `<AlertCircle>`) — never rely on color alone
- Associate error with input via `aria-describedby={errorId}` and `aria-invalid="true"`

```tsx
<Input
  id="email"
  aria-invalid={!!error}
  aria-describedby={error ? 'email-error' : undefined}
  className={cn(error && 'border-danger focus-visible:border-danger')}
/>
{error && (
  <p id="email-error" className="flex items-center gap-1.5 text-sm text-danger">
    <AlertCircle className="h-4 w-4" /> {error}
  </p>
)}
```

### 8.5 Progress indicators

**Course/module progress bar:**
- Height scale (pick by context):
  - `h-1` (4px) — meta rows inside sidebars and sticky footers
  - `h-1.5` (6px) — compact cards (course rail, course grid)
  - `h-2` (8px) — prominent placements (dashboard hero, XP bar)
- `rounded-full`, track `bg-border`, fill `bg-primary`
- **Fill is ALWAYS a solid `bg-primary` — gradients are banned on progress bars.** No `bg-gradient-to-r`, no amber/emerald shifts, no shimmer overlays. Brand as signal, not fireworks.
- Smooth transition on value change: `transition-[width] duration-[400ms] ease-out` — animate only `width`, not `all`, so color/opacity changes stay instant
- Include `role="progressbar"`, `aria-valuenow`, `aria-valuemin`, `aria-valuemax`, `aria-label`
- Render values as rounded integers at display sites (`Math.round(value)`). Stored percentages may be fractional; displayed percentages never are — `66.7%` feels computational, `67%` feels human.

**Completion checkmark:**
- Circular, `bg-success` fill, white checkmark icon
- Size: 20px (inline), 32px (prominent), 48px (celebratory)

**Quiz/assessment segmented meter:**
- Segmented pills: earned = `bg-primary`, unearned = `bg-border`
- 4–10 segments typical; use linear bar if more

### 8.6 Navigation

**Left sidebar (primary nav):**
- Width: 264px expanded, 64px collapsed
- Background: `bg-surface`, border-right `border-border`
- Items: 40px tall, `px-3 py-2 rounded-md`
- Default state: `text-foreground-muted`
- Hover: `bg-surface-muted text-foreground`
- **Active state:**
  - `bg-primary-subtle text-primary font-medium`
  - 3px `bg-primary` accent bar on the left edge (use `before:` pseudo-element or a dedicated span)
- Icons: 20px, inherit text color

```tsx
<Link
```

### 8.12 Media viewers (PDF, video)

PDF and video are embedded documents. They are *inside* our UI but not *of* our UI — mute their native chrome so the app shell stays dominant.

**PDF viewer:**

**Preferred:** Custom `react-pdf` implementation with shadcn-styled toolbar (see §16 Known gaps).

**Acceptable fallback (current implementation):** If an iframe-based PDF viewer must be retained temporarily, the iframe URL MUST include the fragment `#toolbar=0&navpanes=0&scrollbar=0&view=FitH` to suppress the native browser PDF chrome:
- `toolbar=0` — hides pdf.js's top toolbar (zoom, rotate, draw, print, download, save-to-Drive)
- `navpanes=0` — hides the thumbnails / sidebar panes
- `scrollbar=0` — hides the browser scrollbar inside the iframe (Chromium)
- `view=FitH` — fits to width by default so the document meets the reader halfway

Wrap the iframe in `rounded-lg border border-border overflow-hidden` with `border-0` on the iframe itself, and a fixed height (`calc(100vh - 16rem)` today). The frame belongs to us; the document belongs to itself. Provide a download fallback for load failures — never leave the user stranded.

**Never acceptable:** An unmodified iframe exposing Chrome / Firefox / Edge native PDF toolbars (zoom, rotate, draw, print, download, save to Drive, etc.) in production. Every learner should see the lesson content, not a browser file manager.

**Video player:**
- 16:9 (`aspect-video`), `rounded-lg overflow-hidden bg-black`
- Native `<video controls>` is acceptable. When we move to a custom player, playhead + progress fill use `bg-primary`.
- Keyboard controls: space = play/pause, arrows = seek, M = mute.

### 8.13 UI copy & microcopy

Rules for any string that ships to a learner, admin, or course manager — button labels, toasts, empty states, tooltips, modal bodies, email subjects, validation errors, attestation text, log captions, anything rendered to a real human inside the product.

**No em-dashes (`—`) in user-facing text unless absolutely necessary.** Em-dashes read as machine-generated to most readers and clutter short UI copy. Almost every em-dash in a UI string can be rewritten as a period, comma, colon, or parenthetical with no loss of meaning. "Absolutely necessary" means the sentence genuinely cannot be reshaped without losing precision (rare in microcopy, occasionally legitimate in long-form admin help text). When in doubt, rewrite.

| Avoid | Prefer |
|---|---|
| `Acknowledge unlocks in 5:59 — keep this tab open` | `Acknowledge unlocks in 5:59. Keep this tab open.` |
| `Course completed — ask an admin to reset` | `Course completed. Ask an admin to reset.` |
| `Saving — please wait` | `Saving…` |
| `User not found — check the email and retry` | `User not found. Check the email and retry.` |

This rule applies to **user-facing strings only**. Em-dashes remain fine in:
- This design doc and other developer docs.
- Code comments and JSDoc.
- Commit messages and PR bodies.
- `CLAUDE.md`, `AGENTS.md`, handoff docs.

En-dashes (`–`) for numeric ranges (`Mon–Fri`, `Jan–Mar`) are fine. Hyphens (`-`) for compound modifiers (`server-side`, `non-admin`) are fine.

**Other microcopy conventions** (assumed elsewhere, codified here for completeness):
- Sentence case for buttons, headings, and labels — never Title Case. ("Mark complete", not "Mark Complete".)
- No trailing periods on button labels or one-line empty states.
- Numerals for any number meant to be scanned (`6 minutes`, `3 of 12`), spell out only when leading a sentence (`Six policies remain outstanding.`).
- Times are 24-hour with a leading zero if displayed alone (`09:30`); relative ("2 days ago") in feeds.
- Errors lead with the problem, not the apology. ("Couldn't send invite. Resend rate limit hit." NOT "Sorry, we ran into a problem trying to send your invite.")

## 9. Motion & interaction

Motion exists to communicate state change — not to decorate. Every animation in the app earns its place by either (a) confirming that user intent landed, (b) revealing structure as new content arrives, or (c) signaling the system is alive while waiting on data. Anything else is noise. The token, utility, and primitive layers below are the only sanctioned vocabulary; ad-hoc `transition-duration-200` or one-off keyframes should migrate onto them.

### 9.1 Motion tokens

Motion is tokenized like color and spacing. **Never hardcode a duration or easing function in component CSS** — bind to a token so timing stays coherent across the app.

Durations (in `src/app/globals.css`):

| Token | Value | Use |
|---|---|---|
| `--duration-instant` | 100ms | Button press, checkbox flip — too short for the eye to track, just enough to feel responsive |
| `--duration-fast` | 150ms | Hover state changes (background, border, color) — the surface-interactive default |
| `--duration-base` | 200ms | Modal/drawer enter, most micro-transitions |
| `--duration-slow` | 300ms | Lesson thumbnail inner scale, drawer/sheet enter, larger surface morphs |
| `--duration-reveal` | 220ms | One-shot entrance reveals (`<RevealOnView>` opacity + Y rise) |
| `--duration-affirm` | 600ms | Completion-pulse affirmations (progress-bar fill on lesson complete, XP count-up) |
| `--duration-ambient-shimmer` | 1600ms | Skeleton-shimmer loop, progress-bar shimmer highlight |
| `--duration-ambient-breathe` | 8s | Reserved for slow breathing/idle indicators (not currently consumed) |
| `--duration-ambient-drift` | 28s | Hero gradient slow drift |

The four "ambient" tokens are intentionally long. They exist for **background-of-attention** motion — never for anything the user is acting on.

Easing:

| Token | Curve | Use |
|---|---|---|
| `--ease-out-expo` | `cubic-bezier(0.16, 1, 0.3, 1)` | Default for entering / settling — arrives fast, decelerates into rest |
| `--ease-in-out-quart` | `cubic-bezier(0.76, 0, 0.24, 1)` | Loops only (shimmer, breathe, drift) where symmetry is the point |
| `--ease-out-back` | `cubic-bezier(0.34, 1.56, 0.64, 1)` | Affirmation overshoots only — completion pulse, success badge appear. Never on hover |

> **Mandate:** entrances use `--ease-out-expo`. Loops use `--ease-in-out-quart`. Overshoots are reserved for one-shot affirmations; never use `--ease-out-back` on a hover or recurring transition — the springy overshoot reads as bug, not delight, when repeated.

### 9.2 Surface utilities

Two CSS utilities encapsulate the interactive vocabulary so consumers don't repeatedly spell out `transition-colors duration-150 ease-out`.

**`.surface-interactive`** — applied to every clickable surface that isn't a primary button (card, row, badge-as-link, sidebar item). Transitions `background-color, border-color, box-shadow, color` over `--duration-fast` with `--ease-out-expo`. Migrate ad-hoc `transition-colors` strings onto this utility as you touch them.

**`.hover-lift`** — canonical card hover. Strengthens the border, raises to `--shadow-card-md`, and (motion-safe only) translates `-2px` up while promoting the element to its own stacking context via `z-index: 10`. The stacking context is non-negotiable — without it, the lifted top edge slips behind the next sibling card and the lift visually pops in/out as the cursor crosses the gap. Reduced-motion users still receive the border + shadow change, just not the translate.

> **Mandate:** every clickable card uses `.hover-lift` — do not roll bespoke hover patterns with custom transforms or shadows. Lift is the single canonical card-hover affordance.

Shadow tokens supporting the above: `--shadow-card-sm` (resting card) and `--shadow-card-md` (hovered or elevated card). Both have light + dark mode definitions; the dark variants include an inset highlight to lift cards off near-black surfaces.

### 9.3 Motion primitives

Two React primitives encapsulate the journey-reveal patterns. Use these instead of writing IntersectionObserver or `requestAnimationFrame` per component.

**`<RevealOnView>`** (`src/components/ui/RevealOnView.tsx`)

One-shot entrance reveal. Wraps content that should fade + rise into place the first time it crosses the viewport, then disconnects the observer (no continuous scroll coupling — this is explicitly *not* parallax). Stagger sibling reveals with the `delay` prop (typical cadence: 0 / 60 / 120ms).

```tsx
<RevealOnView>
  <NextStepBanner ... />
</RevealOnView>
<RevealOnView delay={60}>
  <DeadlineAlerts ... />
</RevealOnView>
<RevealOnView delay={120}>
  <CourseProgressList ... />
</RevealOnView>
```

Reduced-motion users see fully revealed content on mount — the primitive reads `prefers-reduced-motion` at mount via a lazy initializer, so the very first paint already returns the revealed state. No flicker, no observer ever wires up.

**`<AnimatedNumber>`** (`src/components/ui/AnimatedNumber.tsx`)

Count-up tween for numeric values that should "land" rather than appear. Default duration `--duration-affirm` (600ms), ease-out cubic implemented in JS. Pass `format` to animate formatted strings (durations, percentages) alongside the number — the formatter receives in-flight values, not just the final. Always renders inside a `tabular-nums` span so digits don't shift width as they tick.

```tsx
<AnimatedNumber value={user.xp} />
<AnimatedNumber value={progress} format={(n) => `${Math.round(n)}%`} />
```

`value={null}` renders nothing — use this to suppress the `0` flash while loading. Reduced-motion users skip the tween and receive the final value immediately, gated by an explicit `matchMedia` check (rAF loops aren't covered by the global `transition-duration: 0.01ms` rule — see §9.6).

### 9.4 Journey patterns

Three composed patterns built from the tokens, utilities, and primitives above. These are the only sanctioned multi-element motion patterns in the app.

**Page-load stagger.** Dashboard, course catalog, and admin index pages reveal their primary sections with 0 / 60 / 120ms `<RevealOnView>` cascade. Cap at three steps — staggering five sections at 200ms each ends up feeling like a loading screen, not a reveal. Hero / header is never wrapped (it's already on-screen at first paint).

**Scroll-tick lesson footer.** The lesson-page footer progress track shows two fills: the saved progress (primary) and a secondary fill (`bg-primary/30`) that tracks live scroll depth through the lesson body. The secondary fill is read-only feedback — it does not write progress; the **Mark complete** button still does. Implementation in `src/components/courses/LessonFooter.tsx`.

**Completion affirm pulse.** When a lesson transitions to completed, the footer progress fill runs `.animate-progress-affirm` once — a `scaleY` lift to 1.6× over `--duration-affirm` with a brief box-shadow halo, eased with `--ease-out-back`. This is the canonical "your action landed" affordance for terminal state changes (lesson complete, course complete, ack submitted). Do not stack it with confetti or other reward motion on the same event — pick one.

### 9.5 Course-card thumbnail lift

The course card composes `.hover-lift` (outer geometry) with an inner thumbnail `motion-safe:group-hover:scale-[1.03]` over `duration-slow` with `ease-out-expo`. The two motions are intentionally desynced: the outer lift is fast (`--duration-fast`, 150ms) and the inner image scale is slow (300ms), so the card answers immediately and the image sells the depth a beat later. Don't unify the durations or both motions become a single, blander event.

### 9.6 Reduced motion

Reduced motion is honored in **three layers** because no single layer covers everything:

1. **Global CSS kill-switch** (`@media (prefers-reduced-motion: reduce)` in `globals.css`) — sets `animation: none !important` on every named `.animate-*` class and `transition-duration: 0.01ms` on all selectors. Catches CSS keyframes and transitions.
2. **`motion-safe:` Tailwind variants** — used on `transform`, `transition`, and animation classes that should *only* engage when motion is allowed. Catches utility-class motion the global kill-switch doesn't see (e.g. `motion-safe:hover:scale-[1.03]`).
3. **Explicit JS short-circuit in primitives** — `<RevealOnView>` and `<AnimatedNumber>` read `window.matchMedia("(prefers-reduced-motion: reduce)")` directly and return the final state without ever starting an observer or rAF loop. **This layer is mandatory** for any new JS-driven animation; the global CSS rule does not stop a running `requestAnimationFrame` loop or an IntersectionObserver from firing.

> **Mandate:** when adding a new motion primitive that uses JS (rAF, observers, timers), the reduced-motion check must short-circuit the *JS path*, not just the CSS. Wrapping a tweening loop in a `motion-safe:` class only hides the visual effect — the loop still runs and burns the main thread.

Verify by toggling **System Settings → Accessibility → Reduce motion** on macOS / **Settings → Ease of Access → Display → Show animations** on Windows. All entrances should resolve to their final state on first paint with no visible motion.

### 9.7 Forbidden motion

- Parallax scroll effects (continuous scroll-position coupling — `<RevealOnView>` is one-shot and disconnects, which is the carve-out).
- Scroll-triggered animations on primary content (text reflow, layout shifts).
- Auto-playing decorative animations that loop indefinitely on the primary content area — ambient motion is reserved for backgrounds and skeleton states (see §9.1 ambient tokens).
- Bouncy/elastic easing on recurring transitions (`--ease-out-back` is one-shot affirmations only — see §9.1 mandate).
- Motion longer than 400ms for UI feedback (hover, click, focus). The 600ms `--duration-affirm` is the carve-out for terminal-state celebrations only.
- Hardcoded durations or easing curves in component CSS. Bind to a token from §9.1 — if no token fits, the answer is to add a token, not to write a magic number.

### 9.8 Submit-state buttons (`<FormButton>`)

Every form submit in the app routes through `<FormButton>` (`src/components/ui/FormButton.tsx`). Bespoke `<button type="submit" disabled={saving}>{saving ? "Saving…" : "Save"}</button>` patterns should migrate onto this primitive as you touch them.

States:

- **idle** — renders the button label.
- **pending** — renders an inline `Loader2` spinner + the `pendingLabel` prop (defaults to children + "…"). `disabled` is forced true; `aria-busy="true"` is set.
- **success** — renders an inline `Check` icon + the `successLabel` prop (defaults to children). Held for `successHoldMs` (default 1200ms) then reverts to idle.

```tsx
<FormButton type="submit" loading={isPending} success={justSaved} pendingLabel="Saving…">
  Save signature
</FormButton>
```

Loading detection: inside a `<form>` the button reads React 19's `useFormStatus()` automatically. Outside a form (fetch-based submits), pass `loading={isPending}` explicitly.

The `success` prop is **edge-triggered** — set it `true` once after the mutation resolves and the button manages the hold + revert. Consumer does not need to clear it.

> **Mandate:** new form-submission UI must use `<FormButton>` for the primary action. The success affirmation closes the loop the operator started; without it, the only feedback is a toast that may auto-dismiss before they look back at the form.

### 9.9 Mutation pulse on rows / cards (`useMutationPulse`)

When the user mutates a row in a list (role change, status flip, enrollment toggle), the just-mutated row briefly highlights so they don't lose track of which row updated. This is the canonical "your action landed" affordance on dense surfaces where a toast alone doesn't anchor the change visually.

```tsx
const { pulse, pulseClass } = useMutationPulse();
async function onRoleChange(userId: string, role: Role) {
  await updateUserRole(userId, role);
  pulse(userId);
}
return (
  <tr className={cn("hover:bg-surface-muted", pulseClass(user.id))}>...</tr>
);
```

Implementation: `pulse(id)` adds the `surface-pulse` class for 1100ms (one `--duration-affirm` cycle + decay), then removes it. Backed by the `surfacePulse` keyframe in `globals.css` — accent-tinted background fade in, fade out, eased with `--ease-out-expo`.

**Budget:** one row pulsing at a time on a given table is the norm. Mass-mutation flows (bulk delete, bulk reassign) should pulse only the most-recently-changed row, not all of them — a wall of pulses reads as confusion, not feedback.

### 9.10 List morph on filter / sort (`useListMorph`)

Result-set reorder (filter, sort, search) should cross-fade rather than snap. `useListMorph` wraps `document.startViewTransition` so the browser captures pre-state, runs the state update, captures post-state, and animates the difference.

```tsx
const morph = useListMorph();
const onFilterChange = (next: Filter) => {
  morph(() => setFilter(next));
};
```

Falls through to the callback directly on SSR, on browsers without `startViewTransition` (Firefox today), and when `prefers-reduced-motion: reduce` is set — so callers can use it unconditionally.

> **Mandate:** any state update that reorders or filters a visible list passes through `useListMorph`. Direct `setState` snaps; the morph reads as careful even when the result set is small.

## 10. Accessibility (non-negotiable)

- **WCAG 2.1 AA minimum** across all components. AAA on body text is achievable and preferred.
- **Focus rings use `focus-visible:`, not bare `focus:`.** We only want the keyboard-focus ring on keyboard-focus — showing a full ring on every mouse click pollutes the UI and trains users to ignore it. The canonical string is `focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-ring focus-visible:ring-offset-2 focus-visible:ring-offset-background` (swap `ring-offset-surface` when the button sits on the sidebar). Never remove outlines without replacing.
- **Keyboard navigation:** tab order matches visual order; modals trap focus; escape closes overlays.
- **Color never the only signal:** status badges pair color + icon + text. Error states pair color + icon + message.
- **Semantic HTML:** use `<button>` for actions, `<a>` for navigation, `<h1>-<h6>` for heading hierarchy (one `<h1>` per page).
- **ARIA:** only where semantic HTML isn't enough. Prefer native.
- **Form errors** announce via `aria-describedby` + `aria-invalid`.
- **Icons:** meaningful icons have `aria-label`; decorative icons have `aria-hidden="true"`.
- **Images:** all `<Image>` components have `alt` (empty `alt=""` for decorative).
- **Reduced motion:** respected globally.
- **Skip-to-content link** at the top of every page for keyboard users.

### 10.1 Focus rings: the canonical pattern

All interactive elements use `focus-visible:` utility variants. Never apply `ring-*` utilities outside a `focus-visible:` (or `focus:`) variant. Doing so causes rings to appear at rest on mouse click.

**Canonical pattern:**
```tsx
<button
  className="focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-ring focus-visible:ring-offset-2 focus-visible:ring-offset-background ..."
>
```

Use `ring-offset-surface` instead of `ring-offset-background` when the button sits on the sidebar (`bg-surface`).

**Detection:** A repo-wide search for `ring-` should return only matches prefixed with `focus-visible:` or `focus:`. Any other match is a bug.

- Tailwind CSS v4 (via `@theme` directive)
- shadcn/ui components (Radix UI primitives)
- `next-themes` for dark mode toggle
- `lucide-react` for icons (consistent stroke, tree-shakeable)

### 11.3 Theme provider setup

```tsx
// app/layout.tsx
import { ThemeProvider } from 'next-themes';

export default function RootLayout({ children }) {
  return (
    <html lang="en" suppressHydrationWarning className={`${inter.variable} ${jakarta.variable}`}>
      <body className="bg-background text-foreground font-sans antialiased">
        <ThemeProvider attribute="class" defaultTheme="system" enableSystem>
          {children}
        </ThemeProvider>
      </body>
    </html>
  );
}
```

**After generation:** replace the auto-generated `globals.css` token block with the tokens defined in Section 3.

### 11.5 The `cn()` utility

```ts
// lib/utils.ts
import { type ClassValue, clsx } from 'clsx';
import { twMerge } from 'tailwind-merge';

export function cn(...inputs: ClassValue[]) {
  return twMerge(clsx(inputs));
}
```

## 13. Code conventions

- **No hardcoded colors anywhere.** All colors via tokens. Grep for `#` and `rgb(` in the codebase before committing — any hits are bugs.
- **No hardcoded spacing outside the scale.** No `w-[247px]` unless justified with a comment.
- **Component files use PascalCase** (`CourseCard.tsx`). Utility files use kebab-case (`use-debounce.ts`).
- **One component per file.** Smaller helper components can co-locate if tightly coupled.
- **Props interfaces named `{ComponentName}Props`,** defined above the component.
- **Use `cn()` for conditional classes,** never template literals.
- **Never use `any`.** Use `unknown` or proper types.

## 15. When in doubt

- **Check Section 1's thesis first.** Any decision that violates "brand as signal, not wallpaper" is wrong.
- **Prefer shadcn/ui patterns over custom builds.** The Radix primitives underneath have years of accessibility work.
- **Prefer fewer variants over more.** Three button variants, not seven.
- **Prefer tokens over one-off values.** If a new color is needed, add a token — don't inline.
- **Mature over flashy.** This is a learning platform for working professionals, not a marketing site.

If a design decision isn't covered here, default to what Coursera, Sana, or Linear would do. They are the benchmarks.

## 16. Known gaps

This document describes the target. A few pieces are intentionally behind it, tracked here so contributors know what's real versus aspirational.

- **Custom PDF toolbar.** The lesson player uses an iframe with pdf.js toolbar/nav/scrollbar suppressed (§8.12). The long-term plan is a `react-pdf` implementation with our own controls so the viewer feels native to the app rather than muted-foreign.
- **Video duration in lesson meta.** The lesson title meta row (§8.7.2) labels videos as "Video" without a time estimate. Once we surface duration metadata from the SharePoint proxy, append `· N min` the same way text lessons show reading time.
- **Right rail on lesson player.** The three-column layout in §8.7 documents a notes/transcript rail; the current shell ships two columns. The rail is a deferred feature, not a deleted one — the grid has been kept stable so it can slot in.
- **Quiz lesson completion.** Quiz lessons suppress the Mark-complete button in the footer (§8.7.1). The quiz flow handles completion itself. When we unify flows, the footer becomes the single mutation point for every lesson type.

## 17. Implementation gotchas

Lessons learned from previous iterations. Future contributors should check this list before starting similar work.

### Sticky footers must sit outside the scrolling content

The lesson footer is a `sticky bottom-0` element. It must be rendered either as a direct child of a flex-column parent that also contains the scrolling content area, OR as a child of the scrolling container itself. It must **not** be placed inside the scrolling content as a normal flow element — that makes it appear only at the end of scroll, not pinned to the viewport.

### Brand-subtle backgrounds wrap thumbnails regardless of content

Do not conditionally apply `bg-primary-subtle` only when a thumbnail image is absent. The tinted frame is always present; the image (when available) sits on top of it via `object-contain p-6`. Never use `object-cover` for course thumbnails — it can fill the frame with a white image background and negate the tinted container.

### Ring utilities are focus-visible only

Applying `ring-2 ring-ring` in a default class list causes persistent rings on mouse click. Always prefix with `focus-visible:`. The only exception is the active/pressed state of certain toggle buttons, which should be signaled via background rather than ring.

### PDF iframes need URL fragments to suppress browser chrome

`#toolbar=0&navpanes=0&scrollbar=0&view=FitH` on the `src` URL. Without this, every learner sees Chrome's save-to-Drive button, zoom controls, and rotation buttons in their "lessons." All four parameters are required — omitting `scrollbar=0` leaves a scrollbar inside the iframe on Chromium.

### Hero banners are not brand billboards

A hero banner's background is always a neutral surface (`bg-surface-muted`). The brand accent lives in specific elements within the hero (avatar circles, progress fills, primary CTAs). The moment a hero uses `bg-primary` as its root background, the design thesis has been violated. This is the single most-commonly regressed rule — check it first when reviewing dashboard screenshots.

### Reduced motion must short-circuit the JS path, not just the CSS

The global `prefers-reduced-motion` CSS rule sets `animation: none` and `transition-duration: 0.01ms` — that handles keyframes and CSS transitions. It does **not** stop a `requestAnimationFrame` loop, an IntersectionObserver firing, or a `setTimeout` resolving. Any JS-driven motion (count-up tweens, scroll-reveal observers, staggered timers) must read `matchMedia("(prefers-reduced-motion: reduce)")` explicitly and return the final state without starting the loop. `<RevealOnView>` and `<AnimatedNumber>` are the canonical patterns — copy their lazy-initializer + early-return shape.

### `.hover-lift` needs its own stacking context

The lifted top edge of a card translates `-2px` up. Without `z-index: 10` on `:hover`, that edge renders behind the next sibling card in DOM order and the lift visually pops in/out as the cursor crosses the gap between cards. The stacking-context promotion is baked into `.hover-lift` — never override `z-index` on a hovered card or you'll resurface the bug.

### Outer card lift and inner thumbnail scale are intentionally desynced

Course cards use `--duration-fast` (150ms) on the outer `.hover-lift` and `--duration-slow` (300ms) on the inner thumbnail's `motion-safe:group-hover:scale-[1.03]`. Don't unify them. The fast outer answer says "I noticed your hover"; the slow inner scale sells depth. Equal durations collapse both into a single, blander event.

### Tweening numbers must use `tabular-nums`

`<AnimatedNumber>` wraps its output in `tabular-nums` because proportional digits shift width as the value ticks (the `1` is narrower than the `0`), causing the number's right edge to jitter. Any new numeric tween primitive must include the same — and the parent layout must not horizontally center the number (centering re-introduces the jitter even with `tabular-nums`, since the bounding box still wobbles).

### `.surface-pulse` animates `background-color`, which is not strictly compositor-only

The mutation-pulse class fades a background colour rather than animating a compositor property. This is acceptable at admin-table volumes (one row pulsing at a time, ≤50 rows in viewport) — the browser handles a single background-colour fade without dropping frames. If a perf trace ever flags it (likely under bulk-mutation flows or low-end Android), the fallback is a positioned `::after` overlay animating `opacity` instead. Don't switch preemptively; switch when traced.

### `useListMorph` is client-only — needs a `"use client"` boundary

`document.startViewTransition` is a browser API. Server Components can't call `useListMorph`. Wrap the consumer in a client component (every existing consumer is already `"use client"`). If you find yourself wanting list-morph behaviour on a Server Component, lift the morph-relevant state into a client child rather than trying to call the hook server-side.

### `<FormButton>` success is edge-triggered, not level

Setting `success={true}` once kicks the affirmation. Setting it `true` again on the next render (without a transition through `false`) does not re-kick — the button latches on the rising edge. Consumer code should treat `success` as a transient signal: flip it true after the mutation resolves, don't bother flipping it back. If your code accidentally renders `success={true}` permanently after the first success, every subsequent render through-the-component is a no-op (the button stays in its post-hold idle state). If you need to re-affirm the same success state, flip through `false` first.
