# Rollback Procedures

This runbook covers rolling back a failed deployment quickly and safely.

## Decision Criteria

Roll back immediately if you observe:
- [ ] Error rate exceeds 5% for more than 2 minutes
- [ ] P99 latency doubles or exceeds 2 seconds
- [ ] Critical user-facing feature is broken
- [ ] Data integrity issue detected

## Step 1 — Identify the Bad Commit

```bash
# View recent deployments
git log --oneline -10 origin/main

# Check which version is currently deployed
curl -s https://app.teamssquared.com/api/health | jq .version
```

## Step 2 — Trigger Rollback via Render

```bash
# Using Render CLI
render deploys list --service <SERVICE_ID>
render deploys rollback --service <SERVICE_ID> --deploy <PREVIOUS_DEPLOY_ID>
```

Or via the Render dashboard:
1. Navigate to **Services → [service name] → Deploys**
2. Find the last known-good deploy
3. Click **Rollback to this deploy**

## Step 3 — Verify Recovery

```bash
# Poll health endpoint until healthy
watch -n5 'curl -s https://app.teamssquared.com/api/health | jq .'

# Check error rates in logs
render logs --service <SERVICE_ID> --tail 100 | grep -E "ERROR|5[0-9]{2}"
```

## Step 4 — Database Rollback (if needed)

> ⚠️ Only if the deployment included a schema migration.

```bash
# Identify the migration to revert
npx prisma migrate status

# Revert last migration (creates a new migration)
# NOTE: Never use migrate reset in production
npx prisma db execute --file ./migrations/rollback_<name>.sql
```

## Step 5 — Post-Incident

Once stable:
1. Create a Linear ticket tagged `incident` with timeline
2. Write a brief postmortem in Notion within 48 hours
3. Add a regression test for the broken behaviour

## Escalation

| Severity | Escalate to |
|----------|-------------|
| P0 — site down | On-call engineer + CTO |
| P1 — major feature broken | On-call engineer |
| P2 — degraded performance | Team lead |
